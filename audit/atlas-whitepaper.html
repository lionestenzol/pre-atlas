01What this system is

Pre Atlas is one machine for governing your own behavior. At its center is delta-kernel — a deterministic state engine that runs your life as a six-mode lifecycle (RECOVER → CLOSURE → MAINTENANCE → BUILD → COMPOUND → SCALE). Around it, sensors read your conversations, a triage core decides what matters, an execution arm acts on it, and a handful of surfaces let you see and steer the whole thing. 35 subsystems, but one loop.

The architecture is real

Hub-and-spoke around a SQLite state engine, one-way HTTP seams. Every dependency edge in the repo terminates at delta-kernel. ~80% of the spine is wired and several seams are proven end-to-end.

The gap is activation, not code

The "runs but doesn't do" feeling is accurate: the act-loop exists end-to-end but is unscheduled behind three default-off gates. Two env vars + one cron entry would light it. Today only the kernel's own daemon fires.

The arc: islands → country

The ongoing work is making droplist · cycleboard · inPACT · lattice into one item backbone seen through many windows. Unified read, a consuming surface, and write-through are shipped; cross-surface identity is the last brick.

02The shape — one loop, many parts

Forget the folders. Functionally, the 35 subsystems play roles in a single loop. Information flows left to right and feeds back: your conversations are sensed, the result is sorted into decisions, decisions are acted on, and everything is shown on the surfaces you look at — all routed through the one hub.

🧭 The Hub — delta-kernel

The deterministic core every stage routes through. Holds the canonical mode/state, the signals store, and the viewmodels the surfaces render. A governance daemon fires 9 cron jobs (a */5 heartbeat + preparation). If this stops, ten subsystems lose their hub.

05The spine — verified relationships

First-hand code-recon (2026-06-21/22, no subagents — they confabulate causal stories). The real topology is not the arrow-sketch. Each seam is cited to file:line and tagged live · dormant · gap-closed · busted.

droplist → delta-kernel → lattice the real "to lattice" path

droplist graph_engine.py:236 → POST /api/signals/ingest → kernel signals-store → lattice-projection.ts:204 → apps/lattice viewmodel. Live-proven e2e (7640010): drop → DAG settle → authed ingest (202) → lattice items with provenance=droplist.

intake chainer → packet / DAG

droplist server.py:148 POST /api/drop → intake.py:33 → engine.process_drop → settles the secured packet. The bouncer + chainer that turns a raw drop into a runnable graph.

inPACT ↔ cycleboard ↔ kernel

Shared state via kernel /api/cycleboard (inpact/cli/inpact.ts:96). Cycle Board is a view, not an engine — a localStorage SPA syncing the kernel, zero realtime.

item backbone — unified read + write

atlas-map-api GET /items aggregates droplist DAGs + cycleboard cards + inPACT projects into one shape. 373 items live. lattice consumes it; write-through flips droplist status from lattice. Last brick: shared identity across surfaces.

auto-triage → droplist the one real gap — now bridged

Was the only missing rung. Built cognitive-sensor/droplist_bridge.py + Rung 4 in auto_triage (6/6 tests). Dormant until DROPLIST_DROP_URL is set. Uncommitted.

the act-loop (intake→sort→route→close)

Python arm auto_triage → cortex_bridge → Cortex → decisions_to_atlas is intact in code but unscheduled, behind 3 stacked off-gates. The every-5-min popup is the TS governance_daemon.ts — a different engine, same primitive.

the Supabase Realtime pipeline

The "4-part stateless pipeline on a Supabase bus" does not exist. Reality = SQLite hub-and-spoke, one-way HTTP. Supabase lives only in the other repo (pre-atlas\services\delta-scp, hyphen). The two-repo split is load-bearing.

atlas-map — the GPS substrate

atlas-map-api :3072 + atlas-cli + an MCP serve the same system-index.json that built this page. It reads everything (so it shows as an "orphan" in the import graph — its links are read-time HTTP). POST guarded by X-Atlas-Token.

To make the loop live: set DROPLIST_DROP_URL (triage→droplist) + DROPLIST_ATLAS_SIGNALS_URL (droplist→lattice). Then flip AUTO_TRIAGE_APPLY in DRY-RUN first, watch proposals accrue a week, then enable the ARCHIVE-only execute arm last and watched.
